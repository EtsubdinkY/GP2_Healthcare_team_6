# MongoDB Schema Notes - Part A

This file is copy-ready content for the MongoDB schema section of `docs/polyglot_design.pdf`.

## Collection: clinical_notes

**Purpose:** Store clinical documentation with flexible structures across progress notes, consultations, discharge summaries, and procedure notes.

**Document Structure:**
- `_id`: ObjectId, auto-generated by MongoDB
- `patient_id`: Integer, references PostgreSQL `patient.patient_id`
- `provider_id`: Integer, references PostgreSQL `provider.provider_id`
- `encounter_date`: Date of the clinical encounter
- `note_type`: String enum - `progress_note`, `consultation`, `discharge_summary`, `procedure_note`
- `icd10_codes`: Array of diagnosis codes used for clinical filtering
- `signed`: Boolean indicating whether the note is finalized
- `signed_date`: Date when the note was finalized, or null for unsigned notes
- Progress-note fields: `chief_complaint`, `review_of_systems`, `physical_exam`, `assessment`, `plan`
- Consultation fields: `requesting_provider_id`, `reason_for_consultation`, `findings`, `recommendations`, `urgency`
- Discharge-summary fields: `admission_date`, `discharge_date`, `discharge_diagnosis`, `hospital_course`, `discharge_medications`, `follow_up`
- Procedure-note fields: `procedure_name`, `indication`, `anesthesia`, `findings`, `complications`

**Embedding Rationale:** Review of systems, physical exam, vitals, discharge medication names, and procedure details are embedded because they are bounded and are normally viewed together with the note. Patient and provider data are referenced because they already exist as strongly consistent PostgreSQL records.

**Indexes:**
- `{ patient_id: 1, encounter_date: -1 }` for patient timeline lookup
- `{ provider_id: 1, note_type: 1, encounter_date: -1 }` for provider documentation workload
- `{ icd10_codes: 1 }` for diagnosis-code filtering
- Text index on `chief_complaint`, `assessment`, `plan`, `findings`, `recommendations`, `discharge_diagnosis`
- Partial index on unsigned notes for provider review workflow

## Collection: medical_images_metadata

**Purpose:** Store imaging study metadata and radiologist reports while allowing modality-specific DICOM fields.

**Document Structure:**
- `_id`: ObjectId
- `patient_id`: Integer, references PostgreSQL `patient.patient_id`
- `provider_id`: Integer, ordering or associated provider
- `study_date`: Date of imaging study
- `modality`: String enum - `CT`, `MRI`, `XRAY`, `US`, `MAMMO`
- `body_part`: Study body region
- `study_description`: Human-readable study description
- `dicom_metadata`: Embedded object for modality-specific fields such as `study_uid`, `series_count`, `image_count`, `slice_thickness`, `sequence`, `views`, `birads`
- `radiologist_report`: Embedded object containing `provider_id`, `findings`, `impression`, `critical_finding`, and `signed_date`

**Embedding Rationale:** DICOM metadata and radiologist report details are embedded because they are always used with the imaging study and vary by modality. The design avoids forcing CT, MRI, X-ray, ultrasound, and mammography metadata into one rigid SQL table.

**Indexes:**
- `{ patient_id: 1, study_date: -1 }` for patient imaging timeline
- `{ modality: 1, body_part: 1, study_date: -1 }` for imaging-volume analysis
- Unique index on `dicom_metadata.study_uid`
- Partial index on critical findings
- Text index on `study_description`, `radiologist_report.findings`, and `radiologist_report.impression`

## Collection: care_plans

**Purpose:** Store care-management plans with nested goals and interventions.

**Document Structure:**
- `_id`: ObjectId
- `patient_id`: Integer, references PostgreSQL `patient.patient_id`
- `provider_id`: Integer, references PostgreSQL `provider.provider_id`
- `primary_diagnosis`: Embedded object with `name`, `icd10`, and `category`
- `status`: String enum - `active`, `completed`, `paused`, `cancelled`
- `goals`: Array of embedded goal objects with `goal_id`, `description`, `target_date`, `status`, `progress_percent`
- `interventions`: Array of embedded intervention objects with `intervention_id`, `description`, `owner_role`, `frequency`
- `review_frequency_days`: Integer review interval
- `last_reviewed`: Most recent review date

**Embedding Rationale:** Goals and interventions are embedded because each plan has a bounded number of them and care managers normally read or update them as part of the parent care plan. The primary diagnosis is embedded because it supports quick clinical context and still includes the ICD-10 reference.

**Indexes:**
- `{ patient_id: 1, status: 1, last_reviewed: -1 }` for active patient care plans
- `{ provider_id: 1, status: 1 }` for provider workload
- `{ primary_diagnosis.icd10: 1 }` for diagnosis-based filtering
- `{ goals.status: 1 }` for multikey goal-progress tracking
- Text index on diagnosis, goal descriptions, and intervention descriptions

## Collection: patient_surveys

**Purpose:** Store patient-reported outcome surveys with variable question sets and scoring rules.

**Document Structure:**
- `_id`: ObjectId
- `patient_id`: Integer, references PostgreSQL `patient.patient_id`
- `survey_type`: String enum - `PHQ-9`, `GAD-7`, `Pain`, `Satisfaction`
- `completed_at`: Survey completion date
- `responses`: Array of embedded response objects with `question_code`, `prompt`, `score`, `answer_text`
- `score`: Embedded object with `total`, `max_possible`, and `severity`
- `source`: Survey source - `portal`, `tablet`, `phone`, or `paper`
- `notes`: Optional follow-up note

**Embedding Rationale:** Responses are embedded because they are bounded by survey type and always interpreted with the parent survey score. MongoDB is appropriate because PHQ-9, GAD-7, pain, and satisfaction surveys have different structures and may evolve over time.

**Indexes:**
- `{ patient_id: 1, survey_type: 1, completed_at: -1 }` for patient survey timeline
- `{ survey_type: 1, score.severity: 1 }` for risk stratification
- `{ responses.question_code: 1 }` for response-level clinical screening
- Text index on `notes`
